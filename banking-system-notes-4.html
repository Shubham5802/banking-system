1. What is an API Gateway and why do we need one?

The problem without a gateway

Before the gateway, your system had 3 services running on 3 different ports:

Client (Postman / React)
   |
   |--- POST localhost:8081/api/users/register   → user-service
   |--- POST localhost:8082/api/accounts/create  → account-service
   |--- POST localhost:8083/api/transactions/...  → transaction-service

This creates several problems:

  • The client has to know the port of every service
  • JWT validation had to be implemented separately in every service
  • CORS, logging, rate limiting would need to be repeated in every service
  • In production, you cannot expose internal service ports to the internet

What the gateway solves

The gateway is a single entry point. The client only ever talks to port 8080. The gateway decides where to forward the request based on the URL path.

Client (Postman / React)
   |
   └── ALL requests → localhost:8080 (api-gateway)
                           |
                  JwtAuthFilter (validates token)
                           |
              ┌────────────┼────────────┐
              ↓            ↓            ↓
       user-service  account-service  transaction-service
         :8081          :8082            :8083
Analogy: Think of the gateway like a receptionist at a bank. Every visitor (request) walks through the front door (port 8080). The receptionist checks their ID (JWT), then directs them to the right department (service). No visitor walks directly into the back office.

Gateway vs Load Balancer

FeatureAPI GatewayLoad Balancer
Routes by URL pathYesNo
JWT / Auth validationYesNo
Distributes load across instancesSometimesYes (main purpose)
ExampleSpring Cloud Gateway, KongAWS ALB, Nginx

2. Spring Cloud Gateway

Reactive (WebFlux) vs MVC-based gateway

Spring Cloud Gateway comes in two flavours:

ArtifactBased onFilter typeStatus
spring-cloud-starter-gateway-server-webfluxWebFlux (reactive/non-blocking)GlobalFilterCurrent standard
spring-cloud-starter-gateway-server-webmvcSpring MVC (servlet/blocking)HandlerInterceptorNewer, servlet-based

We use the WebFlux based gateway. This means the gateway itself is reactive (runs on Netty, not Tomcat), but the downstream services (user-service, account-service etc.) can still be normal Spring MVC apps. They are completely independent processes — one being reactive does not affect the others.

Why Netty and not Tomcat? Netty is a non-blocking, event-driven network framework. Spring Cloud Gateway uses it because a gateway handles thousands of concurrent connections — blocking one thread per request (Tomcat style) would be wasteful. With Netty, one thread can handle many connections at once.

pom.xml dependency

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway-server-webflux</artifactId>
</dependency>

No version needed — it is managed by the Spring Cloud BOM:

<properties>
    <spring-cloud.version>2025.0.0</spring-cloud.version>
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.cloud</groupId>
            <artifactId>spring-cloud-dependencies</artifactId>
            <version>${spring-cloud.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

application.yaml routing config

server:
  port: 8080

spring:
  application:
    name: api-gateway
  cloud:
    gateway:
      server:
        webflux:
          routes:
            - id: user-service
              uri: ${USER_SERVICE_URL:http://localhost:8081}
              predicates:
                - Path=/api/users/**

            - id: account-service
              uri: ${ACCOUNT_SERVICE_URL:http://localhost:8082}
              predicates:
                - Path=/api/accounts/**

            - id: transaction-service
              uri: ${TRANSACTION_SERVICE_URL:http://localhost:8083}
              predicates:
                - Path=/api/transactions/**

jwt:
  secret: myBankingAppSuperSecretKeyForJWT2024MustBe32Chars

How routes work

Each route has two key parts:

  • predicates — the condition to match. Path=/api/users/** means: any request whose URL starts with /api/users/
  • uri — where to forward the matched request. Uses environment variable with a localhost fallback so the same config works locally and inside Docker
The ${VAR:default} syntax: This is Spring's property placeholder syntax. ${USER_SERVICE_URL:http://localhost:8081} means: use the value of environment variable USER_SERVICE_URL if it exists; otherwise fall back to http://localhost:8081. This is how the same JAR works in both local development and Docker without changing any code.

3. JwtUtil in the Gateway

Why the gateway needs its own JwtUtil

The gateway and user-service are separate JVMs. They do not share code at runtime. The gateway cannot call user-service to ask "is this token valid?" for every request — that would be an extra network call on every single request, adding latency and creating a dependency.

Instead, the gateway has its own copy of JwtUtil that uses the same secret key as user-service to validate tokens independently. Both services share the secret via config, not via code.

What it contains (vs user-service JwtUtil)

Methoduser-service JwtUtilgateway JwtUtil
generateToken()Yes — creates tokens on loginNo — gateway never creates tokens
isTokenValid()YesYes
extractEmail()YesYes
getClaims()Yes (private helper)Yes (private helper)
@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(secret.getBytes());
    }

    public boolean isTokenValid(String token) {
        try {
            getClaims(token).getSubject();
            return true;
        } catch (Exception e) {
            return false;
        }
    }

    private Claims getClaims(String token) {
        return Jwts.parser()
                .verifyWith(getSigningKey())
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }

    public String extractEmail(String token) {
        return getClaims(token).getSubject();
    }
}

4. JwtAuthFilter — GlobalFilter

GlobalFilter vs OncePerRequestFilter

In user-service and account-service, JWT validation uses OncePerRequestFilter — a Spring MVC (servlet) concept. The gateway uses a different mechanism because it is reactive (WebFlux).

OncePerRequestFilterGlobalFilter
Used inSpring MVC servicesSpring Cloud Gateway
Request objectHttpServletRequestServerWebExchange
Response objectHttpServletResponseServerWebExchange
Pass to nextfilterChain.doFilter()chain.filter(exchange)
Return typevoidMono<Void>
Applies toOne serviceAll routes in the gateway
Concept is identical, API is different. Both intercept every request, check the token, and either pass the request forward or block it with 401. The difference is only in the classes used, because one is blocking (servlet) and the other is reactive (WebFlux).

ServerWebExchange explained

In Spring MVC you have two separate objects: HttpServletRequest (incoming) and HttpServletResponse (outgoing). In WebFlux, both are wrapped into a single object called ServerWebExchange.

// Spring MVC equivalent:
HttpServletRequest  request  = ...
HttpServletResponse response = ...

// WebFlux equivalent:
ServerWebExchange exchange = ...
exchange.getRequest()   // the incoming request
exchange.getResponse()  // the outgoing response

Mono<Void> explained

WebFlux is reactive — everything is asynchronous and returns a Mono or Flux. Think of Mono<Void> as a promise that will complete eventually with no return value.

  • chain.filter(exchange) — "pass this request to the next step, I'm done here" (returns Mono<Void>)
  • exchange.getResponse().setComplete() — "send this response now, stop processing" (returns Mono<Void>)

Both return Mono<Void>, which is why the filter method can return either one.

Ordered interface

When multiple GlobalFilter beans exist in the gateway, Spring needs to know which one runs first. The Ordered interface provides a getOrder() method for this.

@Override
public int getOrder() {
    return Ordered.HIGHEST_PRECEDENCE;   // = Integer.MIN_VALUE = runs first
}

Lower number = higher priority = runs earlier. HIGHEST_PRECEDENCE ensures JWT validation runs before any routing or other filter. You do not want a request to be routed to a service before checking whether the caller is authenticated.

Public path exclusion

Register and login do not require a token (the user does not have one yet). These paths must be whitelisted so the filter lets them through without checking the Authorization header.

private static final List<String> PUBLIC_PATHS = List.of(
    "/api/users/register",
    "/api/users/login"
);

Inside the filter, this is checked first:

if (PUBLIC_PATHS.contains(path)) {
    return chain.filter(exchange);   // skip JWT check, pass through
}

Full filter logic walkthrough

@Component
public class JwtAuthFilter implements GlobalFilter, Ordered {

    @Autowired
    private JwtUtil jwtUtil;

    private static final List<String> PUBLIC_PATHS = List.of(
        "/api/users/register",
        "/api/users/login"
    );

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE;
    }

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {

        // Step 1: get the requested URL path
        String path = exchange.getRequest().getURI().getPath();

        // Step 2: if it's a public endpoint, skip JWT check
        if (PUBLIC_PATHS.contains(path)) {
            return chain.filter(exchange);
        }

        // Step 3: read the Authorization header
        String authHeader = exchange.getRequest().getHeaders().getFirst("Authorization");

        // Step 4: header must exist and start with "Bearer "
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // Step 5: extract token (remove "Bearer " prefix = first 7 chars)
        String token = authHeader.substring(7);

        // Step 6: validate the token using JwtUtil
        if (!jwtUtil.isTokenValid(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // Step 7: token is valid, forward the request
        return chain.filter(exchange);
    }
}
Why substring(7)? The Authorization header value looks like: Bearer eyJhbGci.... "Bearer " is 7 characters (6 letters + 1 space). So authHeader.substring(7) gives you just the token part: eyJhbGci...

5. Docker — Full System

Dockerfile for each service

Every Spring Boot service follows the same Dockerfile pattern:

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY target/user-service-0.0.1-SNAPSHOT.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]

Line by line:

  • FROM eclipse-temurin:21-jre — base image containing Java 21 JRE (just the runtime, not the full JDK — smaller image)
  • WORKDIR /app — sets the working directory inside the container to /app
  • COPY target/...jar app.jar — copies the built jar from your machine into the container and renames it app.jar
  • ENTRYPOINT — the command that runs when the container starts; equivalent to java -jar app.jar in terminal
Build order matters: You must run mvn clean package -DskipTests first to produce the jar in the target/ folder, then docker build to package that jar into a Docker image. If you build the image without building the jar first, Docker will fail because the COPY instruction finds no file.

docker-compose.yml — what changed

The original docker-compose.yml had user-service, account-service, transaction-service, MySQL, Redis, Kafka and Zookeeper. Two things were added/fixed:

Environment variable injection

Spring Boot automatically maps environment variables to application properties using a simple rule: uppercase + underscores maps to lowercase + dots.

Environment variable (Docker)application.yaml property
SPRING_DATASOURCE_URLspring.datasource.url
SPRING_DATASOURCE_USERNAMEspring.datasource.username
SPRING_KAFKA_BOOTSTRAP_SERVERSspring.kafka.bootstrap-servers
SPRING_DATA_REDIS_HOSTspring.data.redis.host
USER_SERVICE_URLuser.service.url (or via ${USER_SERVICE_URL})

This means you never change the JAR for different environments. The same image runs locally, in Docker, and in Kubernetes — you only change the environment variables.

restart: on-failure explained

Docker Compose's depends_on only waits for a container to start, not for the service inside it to be ready. MySQL takes 10–15 seconds after the container starts to accept connections. If user-service starts before MySQL is ready, it crashes with a "cannot connect to database" error.

restart: on-failure tells Docker: if this container exits with an error, automatically restart it. By the 2nd or 3rd attempt, MySQL is fully ready and the service starts successfully.

Production-grade fix: The proper solution is a MySQL healthcheck + condition: service_healthy in depends_on. But for development, restart: on-failure achieves the same result with less config.

Docker networking — service name as hostname

When containers run in the same Docker Compose file, Docker creates a shared network between them. On this network, each container is reachable by its service name as the hostname.

# WRONG (works on your machine, fails inside Docker)
SPRING_DATASOURCE_URL: jdbc:mysql://localhost:3306/banking_db

# CORRECT (uses Docker service name as hostname)
SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/banking_db

Inside Docker, localhost refers to the container itself — not to MySQL or any other container. This is the most common mistake when moving from local to Docker.
