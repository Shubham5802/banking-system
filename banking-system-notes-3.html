This DTO bundles three things that come from two different sources:

  • userName and email — fetched from user-service via Feign (UserDto)
  • accounts and totalBalanace — fetched from account-service's own DB

This is a common microservices pattern: API composition — one service aggregates data from multiple sources and returns a unified response.

getAccounts (dashboard) in AccountServiceImpl

public DashboardResponse getAccounts(Integer userId) {
    UserDto user = userClient.getUserById(userId);          // Feign call to user-service
    List<Account> accounts = accountRepo.findByUserId(userId);

    if (accounts.isEmpty()) {
        throw new ResourceNotFoundException("No accounts for: " + userId);
    }

    DashboardResponse response = new DashboardResponse();
    response.setUserName(user.getName());
    response.setEmail(user.getMail());
    response.setAccounts(accounts);

    BigDecimal sum = BigDecimal.valueOf(0);
    for (Account acc : accounts) {
        sum = sum.add(acc.getBalance());
    }
    response.setTotalBalanace(sum.doubleValue());
    return response;
}
Why BigDecimal for money? double has floating-point precision issues. For example: 0.1 + 0.2 in double gives 0.30000000000000004. For money calculations, always use BigDecimal. The final totalBalanace field is stored as Double in the DTO (for JSON simplicity), but the accumulation uses BigDecimal to keep it accurate.

2. Multithreading — @Async in Spring Boot

The problem: why we needed @Async

When createAccount() calls accountEventProducer.publishAccountCreated(), the Kafka publish happens on the same thread that is handling the HTTP request. The client (Postman/frontend) is waiting while Kafka does its work.

Without @Async:
  HTTP request thread ──────────────────────────────────────────▶ response
                        │ create account │ kafka publish │
                                           ↑
                                    client waits here too
                                    (wasted — client doesn't need to)

With @Async:
  HTTP request thread ──────────────────────────────▶ response
                        │ create account │ return
                                          ↑
                        ── banking-async thread: kafka publish
                           (runs in background, client already got response)

The HTTP response returns as soon as the account is saved. The Kafka publish runs on a separate thread from your thread pool — the client doesn't wait for it.

AsyncConfig.java explained

@Configuration
@EnableAsync                    ← tells Spring: this app uses @Async
public class AsyncConfig {

    @Bean(name = "taskExecuter")   ← gives this thread pool a name
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);       ← always keep 5 threads alive
        executor.setMaxPoolSize(10);       ← can grow up to 10 under load
        executor.setQueueCapacity(100);   ← queue up to 100 tasks before rejecting
        executor.setThreadNamePrefix("banking-async");  ← thread names in logs
        executor.initialize();
        return executor;
    }
}

ThreadPoolTaskExecutor parameters explained

ParameterWhat it meansAnalogy
corePoolSize = 55 threads always alive, even if idle5 permanent staff in the office
maxPoolSize = 10Can spin up 5 extra threads under heavy load5 extra contract staff hired during peak
queueCapacity = 100If all 10 threads busy, queue 100 more tasksWaiting room holds 100 people
threadNamePrefixAll threads get this prefix in their nameMakes log output readable — you can see "banking-async-1" vs "http-nio-8082-1"
@EnableAsync must be on a @Configuration class. Without @EnableAsync, the @Async annotation is silently ignored and the method runs synchronously — same thread, no background execution. This is exactly like @EnableCaching being required for @Cacheable.

@Async on AccountEventProducer

@Component
public class AccountEventProducer {

    @Async("taskExecuter")             ← run this method on the named thread pool
    public void publishAccountCreated(String message) {
        System.out.println("publishing on thread: " + Thread.currentThread().getName());
        kafkaTemplate.send("account-created", message);
    }

    // publishEvent() is NOT @Async — used in Saga, needs to be synchronous
    public void publishEvent(String topic, String message) {
        kafkaTemplate.send(topic, message);
    }
}
  • @Async("taskExecuter") — the string matches the @Bean(name = "taskExecuter") in AsyncConfig
  • When called, Spring returns immediately and runs the method body on a background thread
  • The System.out.println will show banking-async-1 (or similar) in logs, confirming it ran on the thread pool, not the HTTP thread
  • publishEvent() is left synchronous intentionally — the Saga needs to wait for debit-success / debit-failed before proceeding
@Async only works when called from OUTSIDE the same class. Spring uses a proxy to intercept @Async calls. If method A in AccountEventProducer calls @Async method B in the same class, the proxy is bypassed and B runs synchronously. This is the same proxy limitation as @Transactional.

3. transaction-service (new microservice)

Why a separate service for transactions?

You could add transfer logic to account-service. But microservices design says: each service should have one clear responsibility.

  • account-service — owns accounts: create, read, update balances
  • transaction-service — owns transfers: initiate, track status, history

This also means the Transaction table is in transaction-service's database, and only transaction-service writes to it. account-service only updates balances.

Transaction.java (entity)

@Entity
@Data
@EntityListeners(AuditingEntityListener.class)
public class Transaction {
    @Id @GeneratedValue
    private Integer id;

    @NotNull
    private String fromAccountId;    ← account number of sender

    @NotNull
    private String toAccountId;      ← account number of receiver

    @NotNull
    private Double amount;

    @Enumerated(EnumType.STRING)
    private TransactionTypes type;   ← TRANSFER (enum, stored as string in DB)

    private String status;           ← PENDING → COMPLETED or FAILED

    @CreatedDate
    private LocalDateTime timestamp = LocalDateTime.now();
}
Status lifecycle: A transaction always starts as PENDING. It only moves to COMPLETED or FAILED once the Saga finishes. The client gets a 202 ACCEPTED response when the transfer is initiated, not when it is complete — the actual result comes asynchronously via Kafka.

202 ACCEPTED means: "I received your request and started processing it, but I haven't finished yet." This is correct for async Saga flows where the actual debit/credit happens in the background via Kafka.

transaction-service calls account-service synchronously (via Feign) just to validate that both accounts exist and that the sender has enough balance. The actual debit/credit is done asynchronously via Kafka (Saga).

TransactionServiceImpl.java — initiateTransfer

public TransactionResponse initiateTransfer(TransferRequest request) {
    // Step 1: Validate both accounts exist via Feign
    Account fromAccount = accountClient.getAccount(request.getFromAccountNumber());
    Account toAccount   = accountClient.getAccount(request.getToAccountNumber());

    // Step 2: Check sufficient balance (early guard, before DB write)
    if (fromAccount.getBalance().doubleValue() < request.getAmount()) {
        throw new RuntimeException("Insufficient Balance");
    }

    // Step 3: Save transaction as PENDING in DB
    Transaction transaction = new Transaction();
    transaction.setFromAccountId(request.getFromAccountNumber());
    transaction.setToAccountId(request.getToAccountNumber());
    transaction.setAmount(request.getAmount());
    transaction.setType(TransactionTypes.TRANSFER);
    transaction.setStatus("PENDING");
    transactionRepo.save(transaction);

    // Step 4: Kick off Saga — publish debit request to Kafka
    String debitMsg = "transactionId=" + transaction.getId()
                    + ",fromAccount=" + request.getFromAccountNumber()
                    + ",amount=" + request.getAmount();
    producer.publishDebitRequest(debitMsg);

    // Step 5: Return PENDING response immediately (202)
    return mapToResponse(transaction);
}

4. Saga Pattern

The problem: distributed transactions

In a monolith, a money transfer is a single DB transaction:

BEGIN TRANSACTION
  UPDATE accounts SET balance = balance - 500 WHERE id = 'A'
  UPDATE accounts SET balance = balance + 500 WHERE id = 'B'
COMMIT

If step 2 fails, step 1 is rolled back automatically. No inconsistency.

In microservices, accounts are in account-service's DB. There is no single transaction that spans two services.

Problem scenario:
  1. Debit 500 from account A ← succeeds
  2. Credit 500 to account B  ← network failure, never happens

Result: 500 vanishes from the system. Money is lost.

This is the distributed transaction problem. Saga is the standard solution.

What Saga is

A Saga is a sequence of local transactions, where each step publishes an event. If a step fails, a compensating transaction (undo action) is triggered for every step that already succeeded.

Happy path:     debit A → credit B → done ✓
Failure path:   debit A → credit B fails → compensate: refund A ✓

Choreography vs Orchestration

There are two ways to implement a Saga. This project uses choreography.

Choreography (what we built)Orchestration (alternative)
How it worksEach service reacts to events and publishes the next event. No central controller.A central "orchestrator" service tells each service what to do step by step.
CommunicationServices talk only via Kafka eventsOrchestrator calls each service
CouplingLoose — services only know about topicsTighter — orchestrator knows all steps
ComplexityLogic spread across services (harder to trace)Logic centralized (easier to trace)
Best forSimple flows with few stepsComplex flows with many steps or conditions

Saga Happy Path — step by step

Postman: POST /api/transactions/transfer
  { "fromAccountNumber": "ABC123", "toAccountNumber": "XYZ789", "amount": 500 }

Step 1 — TransactionServiceImpl.initiateTransfer()
  → Feign: validate both accounts exist (GET /api/accounts/account/ABC123, XYZ789)
  → Check: ABC123 balance >= 500? YES
  → Save Transaction(id=42, status=PENDING) to DB
  → Kafka publish: topic="debit-request"
      message: "transactionId=42,fromAccount=ABC123,amount=500.0"
  → Return 202 ACCEPTED with Transaction(id=42, status=PENDING)

Step 2 — SagaEventConsumer.handleDebitRequest() [in account-service]
  → Parse message: transactionId=42, fromAccount=ABC123, amount=500
  → Load account ABC123 from DB
  → balance (1000) >= 500? YES
  → account.balance = 1000 - 500 = 500
  → Save account
  → Kafka publish: topic="debit-success"
      message: "transactionId=42"

Step 3 — TransactionEventConsumer.handleDebitSuccess() [in transaction-service]
  → Parse transactionId=42
  → Load transaction(42) from DB
  → Get toAccountId = XYZ789, amount = 500
  → Kafka publish: topic="credit-request"
      message: "transactionId=42,toAccount=XYZ789,amount=500.0"

Step 4 — SagaEventConsumer.handleCreditRequest() [in account-service]
  → Parse message: transactionId=42, toAccount=XYZ789, amount=500
  → Load account XYZ789 from DB
  → account.balance = current + 500
  → Save account
  → Kafka publish: topic="credit-success"
      message: "transactionId=42"

Result: ABC123 lost 500, XYZ789 gained 500. Transfer complete.
(Transaction is currently left as PENDING — marking it COMPLETED on credit-success
is a natural next step to add.)

Saga Failure Path — compensation (rollback)

Same scenario, but credit step fails (e.g., XYZ789 account doesn't exist):

Steps 1-3 same as above. ABC123 has already been debited.

Step 4 — SagaEventConsumer.handleCreditRequest() [in account-service]
  → Load account XYZ789 from DB
  → account == null? YES (doesn't exist)
  → Kafka publish: topic="credit-failed"
      message: "transactionId=42"

Step 5 — TransactionEventConsumer.handleCreditFailed() [in transaction-service]
  → Parse transactionId=42
  → Load transaction(42), set status="FAILED", save
  → Kafka publish: topic="compensate-debit"
      message: "transactionId=42,fromAccount=ABC123,amount=500.0"

Step 6 — SagaEventConsumer.handleCompensateDebit() [in account-service]
  → Parse fromAccount=ABC123, amount=500
  → Load account ABC123
  → account.balance = 500 + 500 = 1000  ← money refunded!
  → Save account

Result: ABC123 gets their 500 back. Transaction marked FAILED. No money lost.
Compensation is not a rollback. It is a new forward transaction that undoes the effect. If the compensation step itself fails (network error after debit but before compensation), money can still be lost. In production systems, compensation failures are logged to a "dead letter queue" and resolved manually or retried. Our system currently logs "Compensation failed: " + message to System.err as a placeholder.

TransactionEventProducer.java (transaction-service)

@Component
public class TransactionEventProducer {

    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void publishDebitRequest(String message) {
        kafkaTemplate.send("debit-request", message);    ← starts the Saga
    }

    public void publishCreditRequest(String message) {
        kafkaTemplate.send("credit-request", message);   ← step 2 of Saga
    }

    public void publishCompensateDebit(String message) {
        kafkaTemplate.send("compensate-debit", message); ← rollback step
    }
}

TransactionEventConsumer.java (transaction-service)

@Component
public class TransactionEventConsumer {

    @KafkaListener(topics = "debit-success", groupId = "transaction-service-group")
    public void handleDebitSuccess(String message) {
        Integer transactionId = extractId(message);
        Transaction t = transactionRepo.findById(transactionId).orElseThrow();

        // debit succeeded — now request credit
        String creditMsg = "transactionId=" + transactionId
                + ",toAccount=" + t.getToAccountId()
                + ",amount=" + t.getAmount();
        producer.publishCreditRequest(creditMsg);
    }

    @KafkaListener(topics = "debit-failed", groupId = "transaction-service-group")
    public void handleDebitFailed(String message) {
        // debit failed before any money moved — just mark FAILED
        Integer transactionId = extractId(message);
        Transaction t = transactionRepo.findById(transactionId).orElseThrow();
        t.setStatus("FAILED");
        transactionRepo.save(t);
    }

    @KafkaListener(topics = "credit-failed", groupId = "transaction-service-group")
    public void handleCreditFailed(String message) {
        // credit failed after debit succeeded — must compensate
        Integer transactionId = extractId(message);
        Transaction t = transactionRepo.findById(transactionId).orElseThrow();
        t.setStatus("FAILED");
        transactionRepo.save(t);

        // publish compensate-debit so account-service refunds sender
        String compensateMsg = "transactionId=" + transactionId
                + ",fromAccount=" + t.getFromAccountId()
                + ",amount=" + t.getAmount();
        producer.publishCompensateDebit(compensateMsg);
    }

    private Integer extractId(String message) {
        // parses "transactionId=42,..." → 42
        String part = message.split(",")[0];
        return Integer.parseInt(part.split("=")[1]);
    }
}
Why transaction-service uses the DB in event handlers: When handleDebitSuccess fires, it needs to know the toAccountId and amount for the credit step. It gets this by loading the transaction from its own DB using the transactionId carried in the message. This is how services pass context through Kafka without bloating the message — carry just the ID, look up details locally.

SagaEventConsumer.java (account-service)

@Component
public class SagaEventConsumer {

    @KafkaListener(topics = "debit-request", groupId = "account-service-group")
    public void handleDebitRequest(String message) {
        // message: "transactionId=42,fromAccount=ABC123,amount=500.0"
        try {
            String transactionId = extract(message, "transactionId");
            String accountNumber = extract(message, "fromAccount");
            BigDecimal amount    = new BigDecimal(extract(message, "amount"));

            Account account = accountRepo.findByAccountNumber(accountNumber);

            if (account == null || account.getBalance().compareTo(amount) < 0) {
                producer.publishEvent("debit-failed", "transactionId=" + transactionId);
                return;
            }

            account.setBalance(account.getBalance().subtract(amount));
            accountRepo.save(account);
            producer.publishEvent("debit-success", "transactionId=" + transactionId);

        } catch (Exception e) {
            String transactionId = extract(message, "transactionId");
            producer.publishEvent("debit-failed", "transactionId=" + transactionId);
        }
    }

    @KafkaListener(topics = "credit-request", groupId = "account-service-group")
    public void handleCreditRequest(String message) {
        // message: "transactionId=42,toAccount=XYZ789,amount=500.0"
        try {
            String transactionId = extract(message, "transactionId");
            String accountNumber = extract(message, "toAccount");
            BigDecimal amount    = new BigDecimal(extract(message, "amount"));

            Account account = accountRepo.findByAccountNumber(accountNumber);
            if (account == null) {
                producer.publishEvent("credit-failed", "transactionId=" + transactionId);
                return;
            }

            account.setBalance(account.getBalance().add(amount));
            accountRepo.save(account);
            producer.publishEvent("credit-success", "transactionId=" + transactionId);

        } catch (Exception e) {
            String transactionId = extract(message, "transactionId");
            producer.publishEvent("credit-failed", "transactionId=" + transactionId);
        }
    }

    @KafkaListener(topics = "compensate-debit", groupId = "account-service-group")
    public void handleCompensateDebit(String message) {
        // message: "transactionId=42,fromAccount=ABC123,amount=500.0"
        try {
            String accountNumber = extract(message, "fromAccount");
            BigDecimal amount    = new BigDecimal(extract(message, "amount"));

            Account account = accountRepo.findByAccountNumber(accountNumber);
            if (account != null) {
                account.setBalance(account.getBalance().add(amount));
                accountRepo.save(account);
            }
        } catch (Exception e) {
            // compensation failed — needs manual intervention
            System.err.println("Compensation failed: " + message);
        }
    }

    private String extract(String message, String key) {
        for (String part : message.split(",")) {
            if (part.startsWith(key + "=")) {
                return part.split("=", 2)[1];
            }
        }
        throw new RuntimeException("Key not found in message: " + key);
    }
}
Why try-catch around every handler: If a KafkaListener throws an uncaught exception, Spring Kafka retries the message by default. In a Saga, retrying a debit that already succeeded would debit twice. Wrapping in try-catch and publishing "failed" on any exception makes the behavior predictable: exactly one outcome (success or failure) per message.

6. Common Errors and Fixes (Phase 3)

ErrorCauseFix
@Async method runs on same thread (synchronously) @EnableAsync missing on AsyncConfig, OR method called from within same class Add @EnableAsync to AsyncConfig. Only call @Async methods from outside the class.
Bean named "taskExecuter" not found Typo mismatch between @Bean(name="taskExecuter") and @Async("taskExecuter") Both must use the exact same string. Note: "Executer" not "Executor" in this codebase.
Feign throws 404 when validating account in transaction-service account-service doesn't have GET /api/accounts/account/{accountNumber} yet Ensure account-service has getAccount() method and the endpoint is properly mapped.
Saga debit fires twice (duplicate debit) KafkaListener threw exception, Spring retried the message Wrap the entire listener body in try-catch and publish failure event in the catch block.
Transaction stays PENDING forever credit-success event published but nothing consumes it to mark COMPLETED Add @KafkaListener for "credit-success" in TransactionEventConsumer and set status = "COMPLETED".
message.split(",") breaks if account number contains a comma Simple string parsing is fragile Production fix: use JSON message format (Jackson ObjectMapper). The current approach works for development.
Insufficient Balance thrown even when balance is enough Double comparison: balance < amount on primitives may have floating-point errors Use BigDecimal.compareTo() for balance checks. The Feign-side check in TransactionServiceImpl uses .doubleValue() which is approximate.
transaction-service can't reach account-service account.service.url property not set, or wrong port Check application.yaml: account.service.url: http://localhost:8082. In K8s: http://account-service:8082.
