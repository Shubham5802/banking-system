1. Redis Caching

What is Redis

Redis is an in-memory key-value store. Think of it as a HashMap that lives outside your application as a separate server, survives app restarts, and is shared across all instances of your service.

HashMap (in your app)Redis
Lives inside JVM memoryLives in a separate server/container
Lost when app restartsPersists across app restarts
Only one app instance can use itAll instances share the same cache
No expiry on entriesEntries can have TTL (time to live)

Why Redis in the banking system

Every time account-service calls GET /api/users/{id} via Feign, user-service hits the database. If 100 accounts are created for user ID 1, that's 100 identical DB queries returning the same data.

Without Redis:
  Call 1 → user-service → DB query → return user
  Call 2 → user-service → DB query → return user  (same result, wasted query)
  Call 3 → user-service → DB query → return user  (same result, wasted query)

With Redis:
  Call 1 → user-service → DB query → store in Redis → return user
  Call 2 → user-service → Redis hit → return user   (no DB query)
  Call 3 → user-service → Redis hit → return user   (no DB query)
Redis key format: Spring stores the cached value under key users::1 (cacheName::methodParam). So getUser(1) is cached under users::1, getUser(2) under users::2, etc.

How @Cacheable works step by step

First call — getUser(1):
  1. Spring checks Redis: does key "users::1" exist? → No
  2. Executes the method → hits DB → gets Users object
  3. Stores result in Redis under key "users::1"
  4. Returns the result

Second call — getUser(1):
  1. Spring checks Redis: does key "users::1" exist? → Yes
  2. Returns value from Redis directly
  3. Method body (DB query) is NEVER executed

Changes made to user-service

1. pom.xml — added dependency

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>

Brings in the Lettuce Redis client and Spring Cache Redis integration.

2. application.yml — Redis connection config

spring:
  data:
    redis:
      host: localhost
      port: 6379

Tells Spring where Redis is running. In Docker/K8s this becomes host: redis (service name).

3. UserServiceApplication.java — enable caching

@SpringBootApplication
@EnableJpaAuditing
@EnableCaching    ← without this, @Cacheable is silently ignored
public class UserServiceApplication { ... }

4. UserServiceImpl.java — cache the method

@Cacheable(value = "users", key = "#id")
public Users getUser(Integer id) {
    System.out.println("Fetching user from DB for id: " + id);
    return userRepo.findById(id)
            .orElseThrow(() -> new RuntimeException("User not found"));
}
  • value = "users" — the cache name (prefix of the Redis key)
  • key = "#id" — the method parameter used as the cache key
  • System.out.println — added to visually confirm when DB is hit vs Redis

5. Users.java — implements Serializable

public class Users implements Serializable { ... }

Redis serializes the Java object to bytes to store it. Without Serializable, it throws SerializationException at runtime.

2. Kafka Deep Dive

What Kafka is and why it exists

Kafka is a distributed event streaming platform. In simple terms: a messaging system where one service publishes a message and other services consume it asynchronously.

The problem it solves: Without Kafka, when account-service creates an account, it would need to directly call every other service that cares about this event (notification-service, audit-service, analytics-service). This creates tight coupling.

Without Kafka (tight coupling):
  account-service → calls → notification-service
  account-service → calls → audit-service
  account-service → calls → analytics-service
  (if any of these are down, account creation fails or gets complicated)

With Kafka (loose coupling):
  account-service → publishes to Kafka topic "account-created"
  notification-service → listens to "account-created" independently
  audit-service        → listens to "account-created" independently
  analytics-service    → listens to "account-created" independently
  (account-service doesn't know or care who is listening)

Core Concepts

Topic

A named channel in Kafka. Like a mailbox with a specific name. Example: account-created. Producers send to a topic. Consumers read from a topic. Kafka auto-creates topics on first use.

Producer

The service that sends messages to a topic. In our system: account-service is the producer. Uses KafkaTemplate.send(topic, message).

Consumer

The service that reads messages from a topic. In our system: user-service is the consumer. Uses @KafkaListener annotation. Runs in a background thread automatically.

Consumer Group

A named group of consumers. Kafka tracks which messages each group has read (called offset). If user-service restarts, it resumes from where it left off — it won't miss or re-process messages.

Offset

A number that tracks which message a consumer has read up to. Think of it like a bookmark in a book. auto-offset-reset: earliest means: if this is a new consumer with no bookmark, start from the beginning of the topic.

Feign vs Kafka — when to use which

Feign (synchronous)Kafka (asynchronous)
When to useWhen you need the response to continue (verify user exists before creating account)When you just need to notify and move on (account created, send email)
Caller waits?Yes — waits for responseNo — fire and forget
Consumer down?Request fails immediatelyMessage stored in Kafka until consumer recovers
Use case in bankingValidate user before creating accountNotify other services after account is created

AccountEventProducer.java explained

@Component
public class AccountEventProducer {

    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void publishAccountCreated(String message) {
        kafkaTemplate.send("account-created", message);
    }
}
  • KafkaTemplate<String, String> — generic types are <key type, value type>. We use String for both. Key is used for partitioning (advanced topic).
  • kafkaTemplate.send(topic, message) — sends the message asynchronously. Returns immediately — does not wait for any consumer to receive it.
  • Called in AccountServiceImpl after accountRepo.save()

AccountEventConsumer.java explained

@Component
public class AccountEventConsumer {

    @KafkaListener(topics = "account-created", groupId = "user-service-group")
    public void handleAccountCreated(String message) {
        System.out.println("Account created event received: " + message);
    }
}
  • @KafkaListener — Spring creates a background thread that polls Kafka continuously. When a message arrives on account-created, this method is called automatically.
  • groupId = "user-service-group" — must match the group-id in application.yml. Kafka uses this to track the consumer's offset.
  • Currently just logs to console. In production: send email, write audit log, update analytics.

Kafka Docker setup — two listeners explained

In docker-compose, Kafka needed two listeners:

KAFKA_LISTENERS: INTERNAL://0.0.0.0:9092,EXTERNAL://0.0.0.0:9093
KAFKA_ADVERTISED_LISTENERS: INTERNAL://kafka:9092,EXTERNAL://localhost:9093
  • INTERNAL (9092) — for communication between containers inside Docker network. Spring Boot services use kafka:9092.
  • EXTERNAL (9093) — for access from your host machine (Postman, IntelliJ). Uses localhost:9093.

In Kubernetes, only one listener is needed (PLAINTEXT://kafka:9092) because everything runs inside the cluster — no external access needed.

3. Docker

Image vs Container

Java analogyDocker equivalentDescription
.class fileImageCompiled blueprint — sits on disk, not running
Running JVM processContainerA live running instance of the image
Source codeDockerfileInstructions to build the image

One image can run as many containers as you want. Same way one .class file can be run as many JVM processes.

For MySQL, Kafka, Redis — you pulled pre-built images from Docker Hub (like Maven Central for Docker images). You didn't write a Dockerfile for these.

For your Spring Boot services — nobody on Docker Hub has your code, so you write a Dockerfile and build the image yourself.

Dockerfile explained line by line

FROM eclipse-temurin:21-jre
WORKDIR /app
COPY target/user-service-0.0.1-SNAPSHOT.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
LineWhat it doesWhy
FROM eclipse-temurin:21-jreStart from a base image that has JRE 21You need JRE to run a JAR. JDK is for compiling — Maven already did that.
WORKDIR /appCreate and switch to /app directory inside containerOrganizes files. All following commands run from here.
COPY target/...jar app.jarCopy your built JAR into the containerWithout this, the container has no idea your code exists.
ENTRYPOINT ["java","-jar","app.jar"]Command that runs when container startsExactly what IntelliJ does when you click Run.
Build order: Always run mvn clean package -DskipTests before docker build. Docker copies the JAR from target/ — if you skip Maven, the JAR is stale or missing.

Why localhost doesn't work inside Docker

Inside a container, localhost means the container itself — not your machine. Containers talk to each other using the service name defined in docker-compose.

Wrong:  SPRING_DATASOURCE_URL: jdbc:mysql://localhost:3306/...
Right:  SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/...
                                                ↑
                                    service name from docker-compose

Spring Boot environment variable override

Spring Boot automatically maps environment variables to properties:

Environment variable:  SPRING_DATASOURCE_URL
Maps to property:      spring.datasource.url

Environment variable:  USER_SERVICE_URL
Maps to property:      user.service.url

Rule: Replace _ with . and lowercase. This is how docker-compose env vars override application.yml values without changing the JAR.

imagePullPolicy and allowPublicKeyRetrieval

  • imagePullPolicy: Never — use local image, don't try Docker Hub. Required for locally built images.
  • allowPublicKeyRetrieval=true in MySQL URL — MySQL 8 uses stronger authentication (caching_sha2_password). On first connection without SSL, client needs server's RSA public key. This flag allows it. Fine for development, use SSL in production.

4. Kubernetes

What K8s is and why it exists

Kubernetes is a container orchestration platform. Docker runs containers. Kubernetes manages containers at scale — restarting crashed containers, scaling up/down, load balancing across multiple instances, rolling updates without downtime.

ProblemDocker aloneKubernetes
Container crashesStays down until you manually restartAutomatically restarts
High trafficManually run more containersAuto-scales pods
Deploy new versionDowntime during restartRolling update — zero downtime
Multiple machinesManage each manuallyK8s manages the whole cluster

Cluster, Node, Pod

Kubernetes Cluster
│
├── Node (a machine — physical or VM)
│   ├── Pod → wraps one container (e.g. user-service)
│   ├── Pod → wraps one container (e.g. account-service)
│   ├── Pod → wraps one container (e.g. mysql)
│
└── Node 2 (in production you have multiple nodes)
    ├── Pod
    ├── Pod
K8s conceptDocker equivalentDescription
PodRunning containerSmallest deployable unit. Wraps one container.
Deploymentdocker-compose serviceManages pods — ensures desired number always running
ServiceContainer name + port mappingStable DNS name to reach a set of pods
kubectldocker CLICommand-line tool to interact with K8s cluster
Why you don't manage Pods directly: Pods get random names and IPs that change on every restart. You manage Deployments — K8s manages the pods for you.

Deployment YAML explained

apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service          ← name of this Deployment
spec:
  replicas: 1                 ← how many pods to run (change to 3 for 3 instances)
  selector:
    matchLabels:
      app: user-service       ← Deployment manages pods with this label
  template:
    metadata:
      labels:
        app: user-service     ← label applied to each pod created
    spec:
      containers:
        - name: user-service
          image: user-service:latest
          imagePullPolicy: Never   ← use local image, don't pull from Docker Hub
          ports:
            - containerPort: 8081
          env:
            - name: SPRING_DATASOURCE_URL
              value: "jdbc:mysql://mysql:3306/..."

Why selector and labels must match

The Deployment uses selector.matchLabels to find which pods it owns. The template.labels are the labels applied to pods created by this Deployment. They must match — otherwise the Deployment can't manage its own pods.

Service YAML explained

apiVersion: v1
kind: Service
metadata:
  name: user-service          ← DNS name other pods use to reach this service
spec:
  selector:
    app: user-service         ← routes traffic to pods with this label
  ports:
    - port: 8081              ← port other pods call
      targetPort: 8081        ← port the pod is actually listening on
  type: ClusterIP             ← only reachable inside the cluster

How service discovery works in K8s

When account-service calls http://user-service:8081, K8s DNS resolves user-service to the ClusterIP of the Service, which routes to the correct pod. Same concept as Docker Compose service names, but managed by K8s.

ClusterIP vs NodePort vs LoadBalancer

TypeWho can reach itWhen to use
ClusterIPOnly pods inside the clusterService-to-service communication (user-service → mysql)
NodePortExternal traffic via a port on the nodeDevelopment/testing without a load balancer
LoadBalancerExternal traffic via cloud load balancerProduction on AWS/GCP/Azure

Port forwarding for local testing

Services of type ClusterIP are not accessible from outside the cluster. kubectl port-forward creates a tunnel from your machine to a pod inside K8s.

kubectl port-forward service/user-service 8081:8081
                              ↑                ↑   ↑
                         service name     local  pod
                                          port   port

Result: localhost:8081 → K8s pod running user-service

Every Handling connection for 8081 line you see in the terminal = one request that went through K8s. This is proof K8s is handling it, not Docker.

      | POST /api/accounts (with JWT token)

      |
      | 3. Check Redis cache (users::1)
      |    → Cache HIT: return from Redis (no DB)
      |    → Cache MISS: query MySQL pod, store in Redis, return
      ▼

      |
      | (only on cache miss)
      ▼

      |
      | 4. Save account to MySQL pod
