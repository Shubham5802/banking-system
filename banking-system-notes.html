Key Principle — Microservices: Each service owns its own database table and business logic. Services communicate over HTTP (Feign) for synchronous calls and Kafka for asynchronous events. They are deployed independently and can be scaled independently.

2. Core Concepts Before Code

Why Microservices over your old Monolith?

Monolith (what you knew)Microservices (what you are learning)
One big WAR/JAR deployed togetherMultiple small services, each deployed separately
One database for everythingEach service owns its own data
Scale the whole app even if only one module is slowScale only the slow service
One bug can crash everythingFailure is isolated to one service
All code in one repo / codebaseEach service is an independent project

Why JWT over Sessions?

Session-based (old way)JWT (new way)
Server stores session in DB / memoryServer stores nothing — token is self-contained
Every request does a DB lookup for sessionServer just verifies the signature — no DB hit
Doesn't scale across multiple serversStateless — works across any server
You knew this from JSP sessionsToken sent in Authorization header

Why Spring Data JPA over plain JDBC?

JDBC (what you knew)Spring Data JPA (new way)
PreparedStatement ps = conn.prepareStatement("SELECT ...")userRepo.findById(id)
Write SQL manually for every operationSQL generated from method names
Map ResultSet to object manuallyMapping done automatically
More control, more boilerplateLess boilerplate, convention-based

JWT Structure

eyJhbGciOiJIUzI1NiJ9  .  eyJzdWIiOiJ1c2VyQGdtYWlsLmNvbSJ9  .  abc123signature
        ↓                              ↓                               ↓
    HEADER                         PAYLOAD                        SIGNATURE
  (algorithm)              (email, issued-at, expiry)        HMAC(header+payload, secretKey)

The server never stores the token. On each request it just re-verifies the signature using the secret key. If the token was tampered with, the signature won't match and the request is rejected.

3. user-service Classes

Users.java — Entity

What it is

A JPA entity class that represents a row in the users table in MySQL.

Why it exists

JPA needs a class to know what columns to create and how to map DB rows to Java objects. Without this, you would write raw CREATE TABLE SQL and manage all mappings manually like you did in JDBC.

Key decisions and why

  • Integer id not int — Before saving, JPA sets id to null. Java primitives cannot hold null, so int would cause a NullPointerException.
  • @GeneratedValue(strategy = GenerationType.IDENTITY) — Tells MySQL to auto-increment the id. Without this, you must set the id yourself.
  • @EntityListeners(AuditingEntityListener.class) + @CreatedDate — Spring automatically fills in the creationDate field when you call save(). You never set it manually. Requires @EnableJpaAuditing on the main class.
  • No @NonNull on fields — Lombok's @NonNull adds a null check inside the constructor. Jackson (JSON parser) uses the no-arg constructor when parsing incoming JSON requests, so the null check fires before values are set, causing 400 errors.
@Entity
@Table(name = "users")
@Data
@NoArgsConstructor
@EntityListeners(AuditingEntityListener.class)
public class Users {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer id;
    private String name;
    private String mail;
    private String password;
    @CreatedDate
    private LocalDateTime creationDate;
}

UserRepo.java — Repository

What it is

An interface that gives you database operations without writing SQL.

Why it exists

In old JDBC you wrote SELECT * FROM users WHERE mail = ? and mapped the ResultSet manually. Spring Data JPA reads your method names and generates that SQL automatically.

How it works

@Repository
public interface UserRepo extends JpaRepository<Users, Integer> {
    boolean existsByMail(String mail);       // SELECT COUNT(*) WHERE mail = ?
    Optional<Users> findByMail(String mail); // SELECT * WHERE mail = ?
}

JpaRepository<Users, Integer> — first type is the entity class, second is the primary key type. This gives you save(), findById(), findAll(), deleteById() for free.

CreateUser.java — DTO (Data Transfer Object)

What it is

A plain class representing the JSON body the client sends when registering.

Why it exists

You never expose your Users entity directly as a request/response object because:

  • The entity has id and creationDate which the client should not send
  • The entity stores BCrypt-hashed passwords — you don't want to expose that in responses
  • DTOs let you add validation annotations without polluting the entity
@Data
public class CreateUser {
    @NotBlank(message = "Name is required")
    private String name;
    @NotBlank(message = "Email is required")
    private String mail;
    @NotBlank(message = "Password is required")
    private String password;
}

@NotBlank validation fires when the controller uses @Valid @RequestBody. If the field is blank or null, Spring returns a 400 Bad Request automatically before your service code runs.

LoginRequest.java — DTO

What it is

Holds mail and password for the POST /api/users/login endpoint.

Why it exists

Same DTO separation principle. Login only needs two fields, not the full user structure. Keeping it separate makes the API contract clear.

UserService.java — Interface

What it is

Declares the operations available in the user service layer.

Why it exists

You program to an interface, not a concrete class. This is the same principle you used in Spring MVC. If tomorrow you replace UserServiceImpl with a different implementation, the controller doesn't change. It also makes unit testing easier because you can mock the interface.

Common mistake: Putting @Service on the interface instead of the implementation. Spring tries to register the interface itself as a bean and fails. Always put @Service on the implementation class (UserServiceImpl).

UserServiceImpl.java — Service (Business Logic)

What it is

The actual implementation of the business rules. The brain of user-service.

Why it exists

Controllers should only handle HTTP concerns (read the request, return a response). All business logic lives in the service layer. This separation makes the code testable and maintainable.

Key method flows

save(CreateUser user):
  1. if (userRepo.existsByMail(user.getMail())) throw exception  // prevent duplicate
  2. encode password:  passwordEncoder.encode(user.getPassword())
  3. map DTO fields to Users entity
  4. userRepo.save(entity)
  5. return "User registered successfully"

login(LoginRequest request):
  1. Users user = userRepo.findByMail(request.getMail())
                          .orElseThrow(() -> new exception)
  2. if (!passwordEncoder.matches(request.getPassword(), user.getPassword()))
         throw exception  // wrong password
  3. return jwtUtil.generateToken(user.getMail())

getUser(Integer id):
  1. return userRepo.findById(id)
                    .orElseThrow(() -> new exception)
Why BCrypt? You never store plain passwords. BCrypt is a one-way hashing algorithm. Even if your database is stolen, attackers cannot reverse the hash to get the original password. passwordEncoder.matches(raw, hashed) hashes the raw input and compares — it doesn't decrypt.

JwtUtil.java — Utility

What it is

A Spring component that creates and validates JWT tokens.

Why it exists

After login, you don't want the user to send their username and password on every request. JWT is a signed token issued once at login. On future requests, the server verifies the signature without hitting the database.

How each method works

@Component
public class JwtUtil {
    @Value("${jwt.secret}") private String secret;
    @Value("${jwt.expiration}") private Long expiration;

    // Converts the secret string to a cryptographic key
    private SecretKey getSigningKey() {
        return Keys.hmacShaKeyFor(secret.getBytes());
    }

    // Builds a JWT: subject=email, signed with secret key, expires after 24 hours
    public String generateToken(String email) {
        return Jwts.builder()
            .subject(email)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiration))
            .signWith(getSigningKey())
            .compact();
    }

    // Decodes and verifies the token. Throws exception if expired or tampered.
    private Claims getClaims(String token) {
        return Jwts.parser()
            .verifyWith(getSigningKey())
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    // Reads the email (subject) stored in the token payload
    public String extractEmail(String token) {
        return getClaims(token).getSubject();
    }

    // Returns true if token is valid, false if expired or invalid
    public boolean isTokenValid(String token) {
        try { getClaims(token); return true; }
        catch (Exception e) { return false; }
    }
}

@Value("${jwt.secret}") — Spring injects the value from application.yml. The key must be at the ROOT level in YAML, not nested under spring:.

JwtFilter.java — Security Filter

What it is

A filter that intercepts every HTTP request before it reaches your controller and validates the JWT token.

Why it exists

You need to check the JWT on every protected request. Doing this check inside every controller method would be repetitive and error-prone. A filter does it once, automatically, for all requests.

How it works step by step

Request arrives at server
    ↓
JwtFilter.doFilterInternal() is called
    ↓
Read header: "Authorization: Bearer eyJhbGci..."
    ↓
Strip "Bearer " prefix → get the raw token
    ↓
jwtUtil.isTokenValid(token) → true or false
    ↓
If valid: extract email → create Authentication object
          → set in SecurityContextHolder
          (tells Spring Security: "this request is authenticated as this email")
    ↓
filterChain.doFilter() → pass request to next filter / controller
Why OncePerRequestFilter? In some cases (redirects, forwards), a filter can be called multiple times for a single user request. OncePerRequestFilter guarantees it runs exactly once.

SecurityConfig.java — Security Configuration

What it is

Configures Spring Security: which endpoints are open, which are protected, and how the JWT filter plugs in.

Why it exists

Spring Security blocks ALL endpoints by default when added to the classpath. You must explicitly configure which endpoints are public (/register, /login) and which require authentication.

Key decisions explained

ConfigurationWhy
csrf().disable()CSRF attacks target browser cookie-based sessions. We use JWT in headers — CSRF doesn't apply to us.
sessionManagement(STATELESS)Tells Spring not to create HTTP sessions. JWT is stateless — server remembers nothing between requests.
addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)Run our JWT filter before Spring's default login filter so we handle authentication first.
@Bean PasswordEncoderBCrypt encoder declared as a Spring bean so it can be injected in ServiceImpl and elsewhere.
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Autowired JwtFilter jwtFilter;

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/users/register", "/api/users/login").permitAll()
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);
        return http.build();
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}

AccountEventConsumer.java — Kafka Consumer

What it is

A Kafka listener in user-service that receives events published by account-service.

Why it exists

When account-service creates an account, other services may need to react. Instead of account-service calling those services directly (tight coupling), it publishes an event to Kafka and interested services listen independently.

What it does right now

Currently just logs to the console. In a real system this would trigger a welcome email, SMS notification, audit log entry, etc.

@Component
public class AccountEventConsumer {

    @KafkaListener(topics = "account-created", groupId = "user-service-group")
    public void handleAccountCreated(String message) {
        System.out.println("Account created event received: " + message);
        // TODO: send welcome email, write audit log, etc.
    }
}

@KafkaListener — Spring polls Kafka in a background thread and calls this method whenever a new message arrives on the account-created topic.

4. account-service Classes

Account.java — Entity

What it is

JPA entity representing a row in the account table.

Key decisions and why

  • BigDecimal balance not double — Never use double for money. Floating point arithmetic is imprecise: 0.1 + 0.2 = 0.30000000000000004 in Java. BigDecimal is exact. This is a critical rule in any financial application.
  • Integer userId not @ManyToOne Users join — In microservices, each service owns its own database. account-service does not have the users table, so a JPA join is impossible. You store just the ID and fetch user details via Feign when needed.
  • @Enumerated(EnumType.STRING) — Stores "SAVINGS" or "CURRENT" as text in the DB. If you used EnumType.ORDINAL it stores 0 or 1. If you ever reorder your enum values, ordinal storage would corrupt all existing data. Always use STRING.

AccountType.java — Enum

What it is

Defines the valid account types.

public enum AccountType {
    SAVINGS,
    CURRENT
}

Using an enum instead of a plain String prevents invalid values like "SAVNGS" from being stored. Validation happens at the Java level before it reaches the database.

AccountRepo.java — Repository

Same concept as UserRepo. Extends JpaRepository<Account, Integer> to get free CRUD operations for the Account entity.

CreateAccountRequest.java — DTO

What it is

The incoming request body for POST /api/accounts.

  • userId — which user owns this account
  • initialDeposit — opening balance (BigDecimal)
  • accountType — SAVINGS or CURRENT

UserDto.java — DTO (cross-service)

What it is

A local copy of user fields that account-service needs from user-service.

Why it exists

account-service cannot import user-service's Users entity (they are separate Maven projects with no shared dependency). When Feign calls user-service and gets a JSON response, it automatically maps the JSON fields into this DTO class.

@Data
public class UserDto {
    private Integer id;
    private String name;
    private String mail;
}

AccountService.java — Interface

Declares the service contract. Same purpose as UserService in user-service — program to interfaces for flexibility and testability.

AccountServiceImpl.java — Service

What it is

Business logic for creating a bank account.

Full flow when createAccount() is called

1. userClient.getUserById(request.getUserId())
      → Feign makes HTTP call: GET http://localhost:8081/api/users/{id}
      → If user does not exist: user-service returns 404
        → Feign throws FeignException → account is NOT created
      → If user exists: returns UserDto, execution continues

2. Build Account object:
      → accountNumber = random UUID, first 10 chars, uppercase
      → balance = request.getInitialDeposit()
      → userId = request.getUserId()
      → accountType = request.getAccountType()

3. accountRepo.save(account) → INSERT into MySQL

4. accountEventProducer.publishAccountCreated(message)
      → Sends "userId=X, account=XXXXXXXX" to Kafka topic "account-created"
      → Returns immediately (async) — does not wait for any consumer

5. return "account created"

AccountController.java — Controller

What it is

Handles incoming HTTP requests. Delegates all logic to the service layer.

@RestController
@RequestMapping("/api/accounts")
public class AccountController {
    @Autowired AccountService accountService;

    @PostMapping
    public ResponseEntity<String> createAccount(@Valid @RequestBody CreateAccountRequest request) {
        return ResponseEntity.status(HttpStatus.CREATED).body(accountService.createAccount(request));
    }
}

The controller has no business logic — it only reads the request and returns the response. This keeps it thin and easy to test.

UserClient.java — Feign Client

What it is

An interface that acts as an HTTP client to call user-service endpoints.

Why it exists (comparison with what you knew)

Old way (HttpURLConnection / RestTemplate)Feign way
Write URL, build connection, read stream, parse JSON manuallyDeclare an interface method — Spring generates all the code
20-30 lines of boilerplate3 lines
Error-prone (typos in URL, manual parsing)Type-safe, compile-time checked
@FeignClient(name = "user-service", url = "http://localhost:8081")
public interface UserClient {
    @GetMapping("/api/users/{id}")
    UserDto getUserById(@PathVariable Integer id);
}

@FeignClient tells Spring to generate an HTTP client implementation for this interface. name is a logical name; url is the base URL of user-service. The method mirrors the controller endpoint in user-service exactly.

Version compatibility rule: Spring Cloud (which provides Feign) requires a specific Spring Boot version. Spring Cloud 2025.0.0 requires Spring Boot 3.5.x. This caused hours of debugging — always check the compatibility matrix at start.spring.io before choosing versions.

FeignClientConfig.java — Feign Request Interceptor

What it is

A configuration class that automatically forwards the JWT token from the incoming request to all outgoing Feign calls.

Why it exists

user-service's GET /api/users/{id} is a protected endpoint — it requires a valid JWT token. When account-service calls it via Feign, Feign doesn't attach any headers by default. Without this interceptor, Feign calls return 401 Unauthorized.

How it works

Frontend sends request to account-service:
  Authorization: Bearer eyJhbGci...

FeignClientConfig interceptor runs before every Feign call:
  1. RequestContextHolder.getRequestAttributes()
        → Spring stores the current HTTP request in a thread-local variable
        → This lets you access the request from anywhere (not just the controller)
  2. request.getHeader("Authorization")  → get the JWT from incoming request
  3. template.header("Authorization", authHeader)
        → attach it to the outgoing Feign request

Result: Feign call to user-service includes the JWT automatically

@Configuration
public class FeignClientConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            ServletRequestAttributes attributes =
                (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
            if (attributes != null) {
                String authHeader = attributes.getRequest().getHeader("Authorization");
                if (authHeader != null) {
                    template.header("Authorization", authHeader);
                }
            }
        };
    }
}

AccountEventProducer.java — Kafka Producer

What it is

A Spring component that sends messages to the Kafka topic account-created.

Why it exists

After creating an account, other systems may need to react (send email, update analytics, write audit log). Instead of account-service calling all those systems directly (tight coupling, slow, fragile), it publishes one event to Kafka. Any number of consumers can listen without account-service knowing about them.

Kafka vs Feign — when to use which

Feign (sync)Kafka (async)
When to useWhen you need the response to continue (e.g., verify user exists)When you just need to notify and move on
account-service waits?Yes — waits for user-service responseNo — fires and continues immediately
Failure impactIf user-service is down, account creation failsIf consumer is down, event is stored in Kafka until consumer recovers
@Component
public class AccountEventProducer {
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void publishAccountCreated(String message) {
        kafkaTemplate.send("account-created", message);
    }
}

KafkaTemplate — Spring's Kafka wrapper, same idea as JdbcTemplate wrapping JDBC boilerplate. kafkaTemplate.send(topic, message) is the only line you need to publish an event. Kafka automatically creates the topic on first use.

AccountServiceApplication.java — Main Class

@SpringBootApplication
@EnableJpaAuditing      // enables @CreatedDate on entities
@EnableFeignClients     // scans for @FeignClient interfaces and generates implementations
public class AccountServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(AccountServiceApplication.class, args);
    }
}
  • @EnableJpaAuditing — Without this, @CreatedDate fields are never populated.
  • @EnableFeignClients — Without this, Spring doesn't scan for Feign client interfaces and they won't work.

5. End-to-End Request Flows

Flow 1: User Registration

POST /api/users/register
  Body: { "name": "Shubham", "mail": "s@gmail.com", "password": "pass123" }

1. JwtFilter: no Authorization header → skip, pass through
2. SecurityConfig: /register is permitAll() → allow
3. UserController.save() → calls UserService.save()
4. UserServiceImpl:
     a. userRepo.existsByMail("s@gmail.com") → false → continue
     b. passwordEncoder.encode("pass123") → "$2a$10$xyz..."
     c. Build Users entity, set name, mail, hashed password
     d. userRepo.save(entity) → INSERT INTO users (...)
     e. @CreatedDate fills creationDate automatically
5. Return 201 Created, body: "User registered successfully"

Flow 2: User Login

POST /api/users/login
  Body: { "mail": "s@gmail.com", "password": "pass123" }

1. JwtFilter: no token → skip
2. SecurityConfig: /login is permitAll() → allow
3. UserController.login() → UserServiceImpl.login()
4. UserServiceImpl:
     a. userRepo.findByMail("s@gmail.com") → returns Users entity
     b. passwordEncoder.matches("pass123", "$2a$10$xyz...") → true
     c. jwtUtil.generateToken("s@gmail.com") → "eyJhbGci..."
5. Return 200 OK, body: "eyJhbGci..."   ← this is the JWT token

Flow 3: Create Account

POST /api/accounts
  Header: Authorization: Bearer eyJhbGci...
  Body: { "userId": 1, "initialDeposit": 5000.00, "accountType": "SAVINGS" }

1. JwtFilter:
     a. reads "Bearer eyJhbGci..."
     b. jwtUtil.isTokenValid(token) → true
     c. jwtUtil.extractEmail(token) → "s@gmail.com"
     d. sets SecurityContext → request is authenticated
2. SecurityConfig: authenticated → allow
3. AccountController → AccountServiceImpl.createAccount()
4. FeignClientConfig interceptor: reads Authorization header from current request,
     attaches it to the outgoing Feign call
5. userClient.getUserById(1):
     → GET http://localhost:8081/api/users/1
     → user-service validates the forwarded JWT
     → returns { "id": 1, "name": "Shubham", "mail": "s@gmail.com" }
     → mapped to UserDto
6. Build Account: accountNumber=random, balance=5000, userId=1, type=SAVINGS
7. accountRepo.save(account) → INSERT INTO account (...)
8. accountEventProducer.publishAccountCreated("userId=1, account=XXXXXXXX")
     → KafkaTemplate sends to topic "account-created"
     → Returns immediately, does not wait
9. Return 201 Created, body: "account created"

Meanwhile in user-service (asynchronously):
10. AccountEventConsumer.handleAccountCreated() receives the message
     → prints: "Account created event received: userId=1, account=XXXXXXXX"

6. Key Concepts Summary Table

Concept Old way (JDBC/JSP/Spring MVC) New way (Spring Boot Microservices)
DB access ResultSet rs = stmt.executeQuery(sql) userRepo.findById(id)
SQL writing Manual SQL strings everywhere Method names generate SQL automatically
Password storage Plain text / MD5 BCrypt (one-way hash, cannot be reversed)
Authentication HttpSession in DB or server memory JWT token (stateless, no server storage)
Request filtering Servlet Filter or Interceptor OncePerRequestFilter with SecurityContext
HTTP calls between services HttpURLConnection / RestTemplate boilerplate Feign Client interface (3 lines)
Service-to-service notification Direct method call or REST call (tight coupling) Kafka event (loose coupling, async)
Config values Properties file, manual reading @Value("${key}") auto-injected
Deployment One WAR to one server Multiple services, each in own Docker container
Monetary values Often double (wrong!) BigDecimal (exact, no floating point errors)

7. Common Errors and Fixes (Lessons Learned)

Error Cause Fix
400 Bad Request on register @NonNull on entity fields conflicted with Jackson's no-arg constructor Remove @NonNull from entity fields
Cannot map null into type int JPA sets id to null before save; int can't hold null Change int id to Integer id
Field 'id' doesn't have a default value Table was created before @GeneratedValue was added, so no AUTO_INCREMENT Drop the table and restart (JPA recreates it with AUTO_INCREMENT)
No property 'email' found for type 'Users' Repository method said existsByEmail but field is named mail Rename to existsByMail / findByMail
Cannot resolve 'NotBlank' Missing validation dependency in pom.xml Add spring-boot-starter-validation
jwt.secret not found / null jwt config was nested under spring: in YAML Move jwt config to root level (same level as spring, server)
Encoded password does not look like BCrypt User was registered before BCrypt was wired up — plain password stored in DB Delete the old user row, re-register
Spring Cloud not compatible with Spring Boot Spring Cloud 2025.0.0 requires Boot 3.5.x. Using 3.3.5 fails. Always check compatibility matrix. Used Boot 3.5.13 for account-service.
Feign calls return 401 Feign doesn't forward the JWT automatically Added FeignClientConfig with RequestInterceptor to forward Authorization header
LEADER_NOT_AVAILABLE warnings from Kafka Consumer connected before Kafka finished creating the topic Normal / harmless. Kafka creates the topic and resolves within seconds.
NoSuchBeanDefinitionException for service @Service was on the interface instead of the implementation class Put @Service only on UserServiceImpl, not on UserService interface
